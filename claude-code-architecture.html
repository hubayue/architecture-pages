一、项目概述

1.1 这是什么？

Claude Code 是 Anthropic 公司开发的终端 AI 编程助手 CLI 工具。它在终端中运行，能读取和编辑代码、执行 Shell 命令、搜索代码库、协调多智能体工作流。2026年3月31日，其 npm 包中的 source map 意外暴露了完整的 TypeScript 源码。

1.2 技术栈

组件技术说明
语言TypeScript (strict)全量 strict 模式编译
运行时Bun替代 Node.js 的快速 JS 运行时
终端 UIReact 19 + 自研 Ink用 React 组件渲染终端界面
CLI 框架Commander.js 13命令行参数解析
Schema 验证Zod 3运行时类型校验
API 通信@anthropic-ai/sdk调用 Claude 模型
协议MCP + LSP模型上下文协议、语言服务器协议

1.3 核心数据

  • 40+ 个工具：Bash、文件读写、代码搜索、Web抓取、Agent、MCP工具等
  • ~80 个斜杠命令：/help、/model、/review、/compact 等
  • QueryEngine.ts（1295行）：核心 Agent 回路
  • claude.ts（3419行）：API 请求构建和流式处理
  • main.tsx（4683行）：CLI 入口和模式分发

三、启动流程

3.1 整体流程图

cli.tsx:main() │ ├─ --version / -v ──→ 打印版本号，立即退出（零模块加载） ├─ --dump-system-prompt ──→ 渲染系统提示词 ├─ --daemon-worker ──→ 守护进程工作线程 ├─ daemon ──→ 长运行管理线程 ├─ remote/bridge/sync ──→ 远程会话桥接 ├─ ps/logs/attach ──→ 后台会话管理 │ └─ 默认路径 ──→ await import('../main.js') │ main.tsx:main() │ ├─ preAction 钩子──→ init() → 加载设置、运行迁移 │ ├─ .action() 解析选项 │ │ │ ├─ --print ──→ runHeadless() → QueryEngine │ └─ 默认 ──→ launchRepl() → React/Ink UI │ └─ setup() ├─ 设置 CWD（工作目录） ├─ 创建 Worktree（可选） ├─ 初始化会话记忆 ├─ 预取：命令列表、插件钩子、API密钥 └─ 启动 REPL

3.2 关键设计：快速路径分发

cli.tsx 使用了延迟动态导入策略。像 --version 这样的快速命令不会加载 React、Ink、或者任何工具模块，直接打印版本号退出。

设计要点
默认路径才执行 await import('../main.js')，这意味着只有在真正需要交互式会话时，才会加载 React、Ink 渲染引擎、工具注册表等重型模块。启动性能优化做得非常极致。

3.3 main.tsx 中的模式分发

main.tsx 的 .action() 处理器（约 line 1007）是一个巨大的决策树，根据模式分叉：

  • Headless 模式（--print）：无 UI，使用 QueryEngine 直接输出文本
  • 交互模式（默认）：启动 React/Ink REPL 终端界面
  • 初始化模式（--init）：只运行初始化钩子
  • 远程模式：通过 Bridge 与 IDE/Web 客户端通信

四、查询引擎 — 核心 Agent 回路

4.1 定位

QueryEngine 是整个系统的核心。它管理对话状态、构建系统提示词、处理用户输入、驱动 API 调用、执行工具、输出结果。理解它就理解了 CC 的运作方式。

4.2 核心类结构

class QueryEngine {
  // —— 配置 ——
  private config: QueryEngineConfig  // cwd, tools, commands, MCP客户端, agents...

  // —— 对话状态（核心） ——
  private mutableMessages: Message[]  // 完整对话历史

  // —— 运行时状态 ——
  private abortController: AbortController
  private permissionDenials: SDKPermissionDenial[]
  private totalUsage: NonNullableUsage        // token 用量追踪
  private readFileState: FileStateCache       // 文件读缓存（LRU）

  // —— 记忆 ——
  private discoveredSkillNames: Set<string>
  private loadedNestedMemoryPaths: Set<string>

  // —— 核心方法 ——
  async *submitMessage(userInput)   // 生成器：接收用户输入 → 产出消息流
}

4.3 submitMessage() 的执行步骤

用户输入 prompt │ ▼ 1. 清空技能发现列表、设置 CWD │ ▼ 2. 构建系统提示词 ── fetchSystemPromptParts() （用户上下文 + 系统上下文 + 记忆机制） │ ▼ 3. 处理用户输入 ── processUserInput() （解析斜杠命令 → 构建 ToolUseContext） │ ▼ 4. 记录对话到磁盘（transcript） │ ▼ 5. 更新权限上下文 ── ToolPermissionContext │ ▼ 6. 预取技能和插件（并行） │ ▼ 7. 判断是否应该查询 API ├─ 纯斜杠命令 → 直接返回结果 └─ 需要 AI 推理 → 进入 query() 循环 │ ▼ queryLoop() ← 核心迭代回路

4.4 queryLoop() — Agent 的"心跳"

这是 Agent 的无限循环，每次迭代做以下事情：

  1. 准备工作：从 compact 边界后取消息、应用 tool result budget、压缩历史
  2. 构建系统提示词：拼装完整 system prompt
  3. 自动 Compact 检查：如果 token 数超阈值，调用 compact 服务做上下文总结
  4. 调用 API：通过 deps.makeAPIRequest() 流式获取 LLM 回复
  5. 执行工具：如果模型返回 tool_use 块，调用 runTools() 执行
  6. 停止钩子：调用 handleStopHooks()
  7. 循环或退出：根据 stop_reason、预算、最大轮次决定

4.5 两个关键文件的分工

文件职责类比
QueryEngine.ts管理对话状态、构建提示词、处理用户输入、产出消息流Agent 的"大脑皮层"
query.ts执行 API 调用循环、处理工具结果、上下文压缩Agent 的"神经回路"

五、工具体系

5.1 Tool 接口设计

每个工具都实现 Tool<Input, Output, P> 接口。核心方法如下：

type Tool<Input, Output, P> = {
  // —— 元信息 ——
  name: string                    // 工具名，如 'bash', 'file_edit'
  aliases?: string[]              // 别名
  searchHint?: string             // 搜索提示（3-10个字）

  // —— Schema ——
  inputSchema: Input              // Zod schema → 自动转为 JSON Schema
  outputSchema?: z.ZodType<unknown>

  // —— 核心行为 ——
  call(args, context, canUseTool, parentMessage, onProgress?): Promise<ToolResult>
  description(input, options): Promise<string>

  // —— 权限 ——
  isReadOnly(input): boolean      // 是否只读
  isConcurrencySafe(input): boolean // 可否并发执行
  checkPermissions(input, context): Promise<PermissionResult>

  // —— 渲染 ——
  renderToolUseMessage(input, context): ReactNode   // 工具调用前的UI
  renderToolResultMessage(content, context): ReactNode // 结果的UI

  // —— 启用 ——
  isEnabled(): boolean            // 是否当前可用
}

5.2 buildTool() 工厂函数

位于 Tool.ts:783。为没有显式实现的方法填入安全默认值：

  • isEnabled → () => true
  • isConcurrencySafe → () => false（假设不安全）
  • isReadOnly → () => false（假设会写入）
  • checkPermissions → 返回 { behavior: 'allow' }（交给通用权限系统裁决）
  • userFacingName → () => def.name
设计意图
buildTool() 让每个工具只需实现核心行为（call、description、inputSchema），其余都有合理默认值。新增一个工具最少只需要约 30 行代码。

5.3 工具注册流程

getAllBaseTools()            ← 返回所有内置工具（条件包含）
    │

getTools(permissionContext)  ← 按模式和权限过滤
    │
    ▼

  async call(input, context) {
    // 1. 读取文件
    // 2. 应用替换
    // 3. 写入文件
    // 4. 返回结果
  }

6.3 国产模型适配的关键

CC 使用 ANTHROPIC_BASE_URL 环境变量可以指向任意兼容端点。国产模型厂商（智谱、阿里、火山）已提供 Anthropic 兼容的 API 端点。CCSwitch 工具就是管理这个配置切换的。

七、MCP 集成

7.1 MCP 协议简介

MCP（Model Context Protocol）是一个标准协议，允许外部服务向 AI 模型提供工具。CC 通过 @modelcontextprotocol/sdk 连接 MCP Server，将 MCP 工具透明地注入到工具体系中。

7.2 架构图

CC 工具体系
    │
    ├── 内置工具（Bash, FileEdit, Glob...）
    │
    └── MCP 工具 ── MCPTool（薄壳）
                     │
                     ├── MCP Server 1（Stdio 连接）── tools: [t1, t2, t3]
                     ├── MCP Server 2（HTTP SSE）    ── tools: [t4, t5]
                     └── MCP Server 3（WebSocket）   ── tools: [t6]

7.3 MCPTool 的工作方式

  1. 连接阶段：mcpClient.ts 通过 stdio/HTTP/WebSocket 连接到 MCP Server
  2. 发现阶段：调用 tools/list 获取 Server 提供的工具列表
  3. 注册阶段：为每个 MCP 工具克隆一个 MCPTool 实例，覆盖 name、description、call、inputSchema
  4. 调用阶段：Agent 调用 MCP 工具时，通过 MCP Client 将请求转发给 Server 执行

7.4 这意味着什么

你们要加的 Godot/Unity MCP 工具，只需要写一个 MCP Server（独立的 Python/Node 进程），在 CC 的 MCP 配置中注册它，CC 就会自动将它的工具注入到 Agent 中。完全不需要修改 CC 源码。

开发路线
写 Godot MCP Server → 配置 CC 连接它 → Agent 自动获得游戏引擎工具。CC 源码零改动。

八、权限系统

8.1 权限模式

模式行为
default每个工具调用都弹出交互式权限确认
acceptEdits自动批准只读+编辑，破坏性操作需确认

8.2 权限裁决流程

工具调用请求 │ ▼ 1. 检查拒绝规则 (always-deny rules) ├─ 命中 → 直接拒绝 ✗ └─ 未命中 ↓ │ ▼ 2. 检查允许规则 (always-allow rules) ├─ 命中 → 直接批准 ✓ └─ 未命中 ↓ │ ▼ 3. 分类器自动审批 (仅 Auto 模式的 BashTool) │ ▼ 4. 权限钩子 (PermissionRequest hooks) │ ▼ 5. 交互式权限弹窗（呈现给用户决定）

8.3 关键类型

type PermissionDecision =

  | { behavior: 'deny', reason }    // 拒绝执行
  | { behavior: 'ask' }             // 需要用户决定

type CanUseToolFn = (
  tool: Tool,
  input: Record<string, unknown>,
  toolUseContext: ToolUseContext,
  assistantMessage: AssistantMessage

9.2 性能优化：Memoization

Git 状态查询和 CLAUDE.md 读取使用 lodash-es/memoize 缓存——在同一个会话中，相同的请求不会重复读取磁盘。缓存键是 cwd。

9.3 上下文压缩策略

当对话历史过长时，CC 使用多层次压缩：

  1. Snip Compact：微精简——删除冗余内容
  2. Microcompact：合并相邻的相似消息
  3. Context Collapse：总结中间的对话转为摘要
  4. Auto-Compact：自动检测 token 阈值，触发总结

十一、多智能体协调

11.1 协调者模式

通过 CLAUDE_CODE_COORDINATOR_MODE 环境变量激活。协调者是一个主 Agent，负责：

  • 分析任务 → 拆解为子任务
  • 生成精确的 worker prompt（包含文件路径、行号、完成条件）
  • 分派给 Worker Agent 执行
  • 汇总 Worker 结果 → 生成最终输出

11.2 子 Agent 的三种隔离级别

级别实现用途
Fork Agent进程内 fork，继承父 Agent 的 prompt cache轻量推测、并行搜索
Worktree Agent创建临时的 git worktree，隔离文件系统需要隔离的工作
Background Agent后台异步执行，结果通过消息返回长时间任务

11.3 Fork Agent 的设计精髓

零成本上下文共享

Fork Agent 继承父 Agent 的 prompt cache。因为父子共享相同的系统提示词前缀，子 Agent 的第一次 API 调用几乎可以零成本加载缓存——这是 CC 多智能体系统最关键的性能设计。

11.4 AgentTool 的关键参数

十二、记忆系统

12.1 设计哲学

CC 不使用向量数据库，而是使用文件系统持久化的 Markdown 文件作为记忆存储。这种设计的优点是：人类可读、Git 可追踪、无需额外基础设施。

12.2 四类记忆

类型内容文件示例
User用户角色、偏好、知识背景user_role.md
Feedback用户对 AI 行为的反馈和纠正feedback_testing.md
Project项目状态、决策、约束project_migration.md
Reference外部系统指针（Slack频道、Jira项目等）ref_linear_project.md

12.3 存储位置

~/.claude/projects/<项目slug>/memory/
  ├── MEMORY.md           ← 索引文件（每次对话都加载，最大200行）
  ├── user_role.md        ← 用户角色记忆
  ├── feedback_xxx.md     ← 反馈记忆
  ├── project_xxx.md      ← 项目记忆
  └── reference_xxx.md    ← 引用记忆

12.4 安全机制

  • ENTRYPOINT 截断：MEMORY.md 最多 200 行 / 25KB，超过就截断
  • 漂移保护：如果记忆与当前代码状态冲突，以当前代码为准
  • 5KB+ 自动瘦身：大的记忆文件会被提示压缩
  • "6个月后还有用吗"过滤器：防止记忆膨胀

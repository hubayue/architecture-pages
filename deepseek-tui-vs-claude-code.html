一、定位与技术栈对比

1.1 一句话定位

DeepSeek TUI

面向 DeepSeek V4 模型的终端原生编程 Agent。自包含的 Rust 单二进制，不依赖 Node.js/Python 运行时。MIT 协议开源。

核心理念：把 Agent 作为终端的一个功能——MCP、子 Agent、RLM、持久化任务队列全部内置在一个二进制里。

Claude Code

Anthropic 的终端 AI 编程助手 CLI，面向 Claude 模型。TypeScript + Bun 运行时，React/Ink 渲染终端 UI。

核心理念：Agent 是一个"终端里的 IDE"——完整的工具生态、权限系统、插件体系、多智能体协调，面向商业产品打磨。

💡 第一印象

DeepSeek TUI 是一个人能端起来的代码库。16 个 crate 各司其职，引擎核心在 core/engine.rs（1990行）+ core/engine/turn_loop.rs（2149行），工具注册在 tools/registry.rs（1366行），MCP 在 mcp.rs（4021行）——每个模块边界清晰，新人一周可以吃透核心流程。

Claude Code 是一个商业产品的完整工程——51万行 TypeScript，1900个文件，高度优化的启动路径、40+工具、80+命令、多层权限、遥测系统——学习曲线陡峭，但每一个设计决策都经过了生产环境的打磨。

二、一根线穿到底：一次对话的完整数据流

这是本文的核心章节。跟踪一句真实的用户请求——"把 login.ts 里的空指针异常修一下"——看它在两个系统里分别怎样从输入变成结果。

2.1 宏观流程对比

DeepSeek TUI 请求生命周期
1. 用户输入
TUI -> EngineHandle.tx_op.send(Op::SendMessage)
通过 mpsc channel 进入引擎后台任务

2. 引擎接管
engine.run() 收到 Op -> handle_send_message()
-> 创建 TurnContext -> Event::TurnStarted
-> handle_deepseek_turn() 进入 turn loop

3. 构建请求
刷新系统提示词 -> 检查 prefix-cache
-> MessageRequest (model + messages + system + tools)
-> DeepSeekClient.create_message_stream()

4. 流式消费
tokio::select! 监听：取消信号 / 流事件 / chunk超时
StreamEvent -> Event::Token / ThinkingStarted / ToolCallStarted
-> 通过 tx_event channel 实时推给 TUI

5. 工具执行
tool_use -> 查询 ToolRegistry
-> tool.execute(input, context)
-> 结果注入消息历史 -> 继续循环

6. 回合结束
Event::TurnComplete -> TUI 恢复可输入状态
引擎和 TUI 始终在不同线程，通过 channel 解耦。
Claude Code 请求生命周期
1. 用户输入
Ink UI -> QueryEngine.submitMessage(userInput)
同线程调用，async function* 开始产出消息流

2. 预处理
fetchSystemPromptParts() -- CLAUDE.md + MEMORY.md + Git
-> processUserInput() -- 解析斜杠命令 / @文件引用
-> 写入 transcript 磁盘日志

3. queryLoop 启动
判断是否需要 AI 推理（纯命令可能跳过）
-> 检查是否需 compact
-> 准备消息列表

4. API 调用
claude.ts 构建 Anthropic API 请求
-> anthropic.beta.messages.create({stream: true})
-> SSE 流式解析

5. 工具执行
tool_use -> 权限裁决（规则->AI分类器->用户确认）
-> Tool.call() -> runTools() 并发执行
-> 结果追加到对话历史 -> 继续 loop

6. 回合结束
stop_reason: end_turn -> 退出 queryLoop
-> Ink 重新渲染 -> 用户看到结果
UI 和引擎在同一线程，通过 React 状态驱动。

2.2 架构本质差异：消息驱动 vs 同线程生成器

DeepSeek TUI：Channel 解耦
// 引擎在独立的 tokio 任务中运行
// TUI 只通过 EngineHandle 发送 Op / 接收 Event

// 发送用户消息（TUI 线程）
engine_handle.tx_op.send(Op::SendMessage {
    content: "修一下 login.ts 的空指针",
    mode: AppMode::Agent,
}).await;

// 接收实时事件（TUI 线程）
while let Some(event) = rx_event.recv().await {
    match event {
        Event::Token { text, .. } => // 追加到当前块
        Event::ToolCallStarted { .. } => // 渲染工具卡片
        Event::TurnComplete { .. } => // 恢复输入
    }
}

TUI 永不阻塞。Agent 思考时用户可滚动历史、切换会话、Ctrl+C 取消。

Claude Code：生成器驱动
// QueryEngine 是 async function*
async *submitMessage(userInput: string) {
    const systemParts = await fetchSystemPromptParts();
    const ctx = await processUserInput(userInput);
    yield* this.queryLoop();  // 主循环
    yield { type: 'result', content: finalMessage };
}

// 消费者（Ink 组件）
for await (const msg of engine.submitMessage(input)) {
    // setState -> Ink 重渲染
}

代码写起来像同步逻辑，但 UI 更新嵌在 Agent 循环中。

💡 为什么这个差异重要？

DeepSeek TUI 的 Channel 解耦架构意味着：TUI渲染、输入处理、Agent推理是三个独立任务。天然支持 Agent 思考时滚动历史、Ctrl+C 取消、管理多个会话。

Claude Code 的生成器驱动更简单直接，但用户输入被阻塞直到当前回合完成（CC 对此做了大量优化，含后台子Agent异步执行）。

如果需要在 Agent 工作时保持 UI 完全响应，DeepSeek TUI 架构更友好。

三、Agent 循环：心跳的两种写法

3.1 循环结构对比

DeepSeek TUI · turn_loop.rs (handle_deepseek_turn)
// 主循环：loop + tokio::select!
loop {
    if self.cancel_token.is_cancelled() { break; }
    while let Ok(steer) = self.rx_steer.try_recv() { ... }
    self.refresh_system_prompt(mode);
    if turn.at_max_steps() { break; }

    // 自动 compact + 容量检查
    if should_compact(...) { compact_messages_safe(...); }
    if self.run_capacity_pre_request_checkpoint(...) { continue; }

    // 构建请求 -> 流式发送
    let request = MessageRequest { model, messages, system, tools, ... };
    let stream = client.create_message_stream(request).await;

    // 流式消费（带 steer 检查 + chunk 超时）
    loop {
        tokio::select! {
            _ = cancel_token.cancelled() => break,
            result = timeout(chunk_timeout, stream.next()) => {
                // TextDelta / ToolUse / Thinking -> Event 推送
            }
        }
    }

    // 执行工具
    for tool_use in tool_uses {
        execute_tool_and_stream_result(tool_use, registry, ...);
    }
    if stop_reason != "tool_use" { break; }
}
Claude Code · query.ts (queryLoop)
// while(true) + async/await
async queryLoop() {
    while (true) {
        const messages = this.messagesSinceCompact();
        await this.applyToolResultBudget(messages);
        const systemParts = await this.buildSystemPrompt();

        if (this.shouldAutoCompact()) {
            await this.compactService.compact();
        }

        // API 调用
        const response = await deps.makeAPIRequest({
            model, messages, system, tools, stream: true
        });

        // 流式处理
        for await (const chunk of response) {
            if (chunk.type === 'content_block_delta')
                yield { type: 'text', content: chunk.delta.text };
            if (chunk.type === 'tool_use') toolUses.push(chunk);
        }

        // 执行工具
        if (toolUses.length > 0) {
            const results = await this.runTools(toolUses);
            this.appendResults(results);
            continue;
        }

        await this.handleStopHooks();
        if (stopReason === 'end_turn') break;
    }
}

3.2 关键差异点

差异点DeepSeek TUIClaude Code
取消机制CancellationToken 每步检查——Esc 即时中断AbortController API层取消——需传播到UI
中途输入Steer机制——Agent思考时可注入新指令（rx_steer channel）无中途输入——必须等当前回合完成
容量控制CapacityController——每步前检查风险级别依赖权限系统裁决
透明重试流断开且无内容时自动重发（最多3次）依赖 SDK 重试策略
CheckpointCycle机制——token超阈值自动新周期+briefing多级Compact：Snip->Micro->Collapse->Auto
Prefix-cache主动追踪——每turn检查hash稳定性，TUI显示命中率依赖API层，客户端无显式追踪
🔍 Steer 机制——DeepSeek TUI 的独有武器

Agent 正在流式生成时，用户可以注入新指令。你说"修 login.ts 的空指针"，Agent 开始读文件——你发现它理解错了，不等完成直接输入"不是 login.ts，是 auth.ts"。Agent 收到 steer，下一轮API调用调整方向。Claude Code 做不到。

// 每个 chunk 后检查 rx_steer channel
while let Ok(steer) = self.rx_steer.try_recv() {
    pending_steers.push(steer);

#[async_trait]
pub trait ToolSpec: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    async fn execute(&self, input: Value, context: &ToolContext)
        -> Result<ToolResult, ToolError>;
    fn capabilities(&self) -> Vec<ToolCapability> { vec![] }
    fn approval_requirement(&self, input: &Value)
        -> ApprovalRequirement { ApprovalRequirement::Default }
}

/// 注册表：HashMap + OnceLock 缓存
pub struct ToolRegistry {
    tools: HashMap<String, Arc<dyn ToolSpec>>,
    context: ToolContext,
    api_cache: OnceLock<Vec<Tool>>,  // prefix-cache友好
}

🔍 大结果路由 (Large Output Router) —— 游戏场景杀手功能

    RouteDecision::PassThrough => {}        // 结果小，直接传
    RouteDecision::Synthesise { .. } => {
        workshop_vars.store(handle_name, raw_output);
        return ToolResult::success(summary)
            .with_handle(VarHandle { ... });  // 摘要+句柄
    }
